# Current Architecture Decisions

This section summarizes the architectural decisions that have been formally approved and currently govern the implementation of EAOS.

Only decisions documented through Architecture Decision Records (ADRs) should be considered authoritative.

This section provides a high-level overview rather than replacing the ADR documentation.

---

## Decision Philosophy

Architectural decisions are made according to the following hierarchy:

```text
Business Need

↓

Architecture Analysis

↓

Trade-off Evaluation

↓

ADR

↓

Architecture Review

↓

Approval

↓

Implementation
```

Architecture decisions are intentional and evidence-based.

---

## Approved Architectural Decisions

The following architectural decisions currently define the EAOS implementation.

### AD-001 — Modular Monolith

EAOS is implemented as a Modular Monolith during the initial phases.

This minimizes operational complexity while preserving clear module boundaries.

Future decomposition into distributed services remains possible.

---

### AD-002 — Domain-Driven Design

Business complexity is modeled using Domain-Driven Design (DDD).

Business capabilities define the primary organizational boundaries.

The domain model remains independent of technical infrastructure.

---

### AD-003 — Hexagonal Architecture

External technologies interact with the system exclusively through Ports and Adapters.

The domain core remains isolated from frameworks, databases, and third-party services.

---

### AD-004 — Capability-Oriented Organization

Repository organization follows business capabilities rather than technology stacks.

Business capabilities are treated as stable organizational units.

---

### AD-005 — Repository as Single Source of Truth

All architectural knowledge, documentation, ADRs, engineering standards, and implementation artifacts are maintained within a single version-controlled repository.

Knowledge fragmentation should be avoided.

---

### AD-006 — Architecture as Code

Architecture shall be continuously enforced through executable mechanisms, including:

- Architecture Tests
- Policy as Code
- Static Analysis
- CI Validation
- Fitness Functions

Architecture documentation alone is insufficient.

---

### AD-007 — Policy as Code

Engineering governance shall be automated wherever possible.

Policies should be executable rather than manually enforced.

---

### AD-008 — Adopt → Adapt → Build

Technology selection follows the mandatory order:

```text
Adopt

↓

Adapt

↓

Build
```

Custom implementation is reserved for genuine competitive advantages.

---

### AD-009 — Model Agnostic AI

Artificial Intelligence capabilities remain independent of specific model providers.

All AI integrations must pass through abstraction layers to prevent vendor lock-in.

---

### AD-010 — Evolutionary Architecture

The architecture is expected to evolve continuously.

Evolution must be supported by:

- Evidence
- ADRs
- Architecture Reviews
- Automated Validation
- Continuous Observability

Controlled evolution is preferred over architectural stagnation.
